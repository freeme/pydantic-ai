## 15.1 异常体系

**文件**：`exceptions.py`

pydantic-ai 设计了清晰的异常层级，覆盖控制流、限制、错误、安全等场景。

### 控制流异常（用于工具和钩子内部信号传递）

| 异常 | 用途 |
|------|------|
| `ModelRetry` | 要求模型重试，发送重试提示词 |
| `CallDeferred` | 标记工具调用为延迟执行（deferred tools） |
| `ApprovalRequired` | 标记工具调用需要人工审批（human-in-the-loop）|
| `SkipModelRequest` | 跳过此次模型请求（在钩子中使用）|
| `SkipToolValidation` | 跳过工具参数验证 |
| `SkipToolExecution` | 跳过工具执行（直接注入预设结果）|

### 用户配置错误

| 异常 | 用途 |
|------|------|
| `UserError` | 用户代码配置错误（如类型不匹配、无效参数）|

### 运行时错误

| 异常 | 继承 | 用途 |
|------|------|------|
| `AgentRunError` | `RuntimeError` | Agent 运行时异常的基类 |
| `UsageLimitExceeded` | `AgentRunError` | 超出 Token/请求限制 |
| `ConcurrencyLimitExceeded` | `AgentRunError` | 超出并发限制 |
| `UnexpectedModelBehavior` | `AgentRunError` | 模型返回意外响应 |
| `ContentFilterError` | `UnexpectedModelBehavior` | 内容安全过滤触发 |
| `IncompleteToolCall` | `UnexpectedModelBehavior` | 工具调用不完整 |

### API 错误

| 异常 | 继承 | 用途 |
|------|------|------|
| `ModelAPIError` | `AgentRunError` | 模型 API 调用失败 |
| `ModelHTTPError` | `ModelAPIError` | HTTP 错误（含状态码和请求信息）|

### FallbackModel 专用

## 15.2 UsageLimits — Token 与请求预算控制

**文件**：`usage.py`

class UsageLimits:
    """控制 Agent 运行的资源消耗上限。"""

def check_before_request(self, usage: RunUsage) -> None:

    if request_limit is not None and usage.requests >= request_limit:
        raise UsageLimitExceeded(...)

def check_before_tool_call(self, projected_usage: RunUsage) -> None:

## 15.3 并发限制基础设施

**文件**：`concurrency.py`、`models/concurrency.py`

### AbstractConcurrencyLimiter ABC

```python
class AbstractConcurrencyLimiter(ABC):
    """并发限制器抽象基类，可扩展为分布式实现（如 Redis）。"""

    @abstractmethod
    async def acquire(self, source: str) -> None:
        """获取并发槽，必要时等待。source 用于可观测性标识。"""

    @abstractmethod
    def release(self) -> None:
        """释放并发槽。"""
```

### ConcurrencyLimiter — 内置实现

```python
@dataclass
class ConcurrencyLimit:
    max_running: int        # 最大并发数
    max_queued: int | None  # 最大排队数（None 表示无限）

class ConcurrencyLimiter(AbstractConcurrencyLimiter):
    """基于 anyio 信号量的本地并发限制器，集成 OTel 可观测性。"""
```

### ConcurrencyLimitedModel — 模型级别并发控制

```python
model = ConcurrencyLimitedModel('openai:gpt-5.2', limiter=5)

# 共享并发池
shared_limiter = ConcurrencyLimiter(max_running=10, name='openai-pool')
model1 = ConcurrencyLimitedModel('openai:gpt-4o', limiter=shared_limiter)
model2 = ConcurrencyLimitedModel('openai:gpt-4o-mini', limiter=shared_limiter)
```

## 15.4 HTTP 重试机制

**文件**：`retries.py`

通过 `httpx` 的传输层实现 HTTP 级别的自动重试：

```python
class TenacityTransport(httpx.BaseTransport):
    """基于 tenacity 的同步 HTTP 重试传输层。"""

class AsyncTenacityTransport(httpx.AsyncBaseTransport):
    """基于 tenacity 的异步 HTTP 重试传输层。"""
```

### 重试策略

```python
from tenacity import (
    retry_if_exception_type,
    wait_exponential,
    stop_after_attempt,
)

# 支持 HTTP 429 Retry-After 头部
def wait_retry_after(retry_state) -> float:
    """从 HTTP 响应的 Retry-After 头部计算等待时间。"""
    response = retry_state.outcome.exception().response
    retry_after = response.headers.get('Retry-After')
    if retry_after:
        return float(retry_after)
    return 0.0
```

## 15.8 SkipTool* 异常 — 工具执行钩子

这三个异常专为高级用例设计：

```python
class SkipToolValidation(Exception):
    """在工具执行钩子中抛出，跳过参数验证（直接传递原始 JSON 数据）。"""

class SkipToolExecution(Exception):
    """在工具执行钩子中抛出，跳过工具函数执行（注入预设的工具结果）。
    
    Args:
        result: 直接注入的工具结果
    """

class SkipModelRequest(Exception):
    """在 before_model_request 钩子中抛出，跳过模型调用（注入预设的模型响应）。
    
    Args:
        model_response: 直接注入的模型响应（如缓存命中）
    """
```

应用场景：
- **缓存**：在 `before_model_request` 钩子检查缓存，命中则 `raise SkipModelRequest(cached_response)`
- **安全审计**：在工具执行前拦截特定工具调用
- **测试**：注入固定响应，无需修改工具代码

## 总结

```
安全与基础设施体系
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 分层异常设计：
   控制流信号（ModelRetry、SkipTool*）与错误（AgentRunError）明确区分
   工具作者使用信号异常控制流程，框架使用错误异常表示失败

2. 多维度资源限制：
   UsageLimits 同时控制请求次数和 Token 数量
   ConcurrencyLimiter 控制并行调用数
   两者都在请求前检查（fail fast 原则）

3. 可扩展并发控制：
   AbstractConcurrencyLimiter 允许替换为分布式实现
   ConcurrencyLimitedModel 在模型层面施加限制
   OTel 集成使并发状态可观测

4. HTTP 重试与 Retry-After 支持：
   TenacityTransport 在传输层处理重试
   遵守服务端的 Retry-After 头部，避免 API 禁封

5. 内置工具注册表：
   __init_subclass__ 自动注册，扩展无需修改注册表代码
   BUILTIN_TOOLS_REQUIRING_CONFIG 区分"即用型"和"需配置型"工具

6. 内容安全隔离：
   ContentFilterError 与普通 HTTP 错误分离
   FallbackModel 默认不对内容过滤重试（避免无效重试）
```
