# 四、输出与结果系统

> **核心文件**：`output.py`（公开 API）、`_output.py`（内部实现）、`result.py`（run 模块）、`usage.py`（Token 统计）、`retries.py`（HTTP 重试）

---

## 4.1 输出类型体系

pydantic-ai 的输出系统支持多种方式将模型的回复映射为 Python 类型：

```
OutputSpec（输出规格）
│
├── str                         → TextOutputSchema（纯文本）
├── type[BaseModel]             → ToolOutputSchema（工具调用 + Pydantic 验证）
├── type[TypedDict]             → ToolOutputSchema
├── type[dataclass]             → ToolOutputSchema
├── callable[str] → T           → TextOutputSchema（文本处理函数）
├── BinaryImage                 → ImageOutputSchema（图像输出）
│
├── ToolOutput(MyModel)         → ToolOutputSchema（显式工具输出）
├── NativeOutput(MyModel)       → NativeOutputSchema（模型原生 JSON Schema）
├── PromptedOutput(MyModel)     → PromptedOutputSchema（提示式 JSON）
│
└── list[...上面任意组合...]    → 多输出类型（Union 语义）
```

## 4.2 公开输出 API（output.py）

### OutputMode

```python
OutputMode = Literal['text', 'tool', 'native', 'prompted', 'tool_or_text', 'image', 'auto']
StructuredOutputMode = Literal['tool', 'native', 'prompted']
```

| 模式 | 含义 |
|------|------|
| `'text'` | 纯文本输出（默认） |
| `'tool'` | 通过工具调用返回结构化数据 |
| `'native'` | 模型原生 JSON Schema 约束（如 OpenAI Structured Outputs） |
| `'prompted'` | 通过 prompt 要求 JSON 输出 |
| `'image'` | 图像输出 |
| `'auto'` | 根据 ModelProfile 自动选择 |

### ToolOutput — 工具输出标记

```python
agent = Agent(
    'openai:gpt-5.2',
    output_type=[
        ToolOutput(Fruit, name='return_fruit'),      # 自定义工具名
        ToolOutput(Vehicle, name='return_vehicle'),
    ],
)
```

- 允许多个输出类型（Union 语义）
- 可以自定义工具名和描述
- 支持独立的 max_retries 和 strict 设置

### NativeOutput — 模型原生结构化输出

```python
agent = Agent(
    'openai:gpt-5.2',
    output_type=NativeOutput(
        [Fruit, Vehicle],
        name='Fruit or vehicle',
        description='Return a fruit or vehicle.',
        template=False,  # 禁用 Schema 提示
    ),
)
```

- 只能有一个 NativeOutput（不能与其他输出类型混合）
- 使用模型的原生 JSON Schema 约束，而非工具调用
- 某些模型（如 Gemini）需要在 instructions 中包含 Schema

### PromptedOutput — 提示式结构化输出

```python
agent = Agent(
    'my-model:my-version',
    output_type=PromptedOutput(
        MyModel,
        template="Please output JSON matching this schema:\n{schema}"
    ),
)
```

- 适用于不支持 native/tool 输出的模型
- 通过自定义 prompt 要求模型输出 JSON
- `{schema}` 占位符被替换为 JSON Schema

### TextOutput — 文本处理函数

```python
def process_text(ctx: RunContext[Deps], text: str) -> MyResult:
    return MyResult.from_text(text)

agent = Agent('openai:gpt-5.2', output_type=TextOutput(process_text))
```

## 4.3 内部 OutputSchema 层级（_output.py）

`OutputSchema.build()` 工厂方法根据用户传入的 `output_type` 创建对应的 Schema 实例：

```
OutputSchema (ABC)
├── AutoOutputSchema         → output_mode='auto'，由 Profile 决定具体模式
├── TextOutputSchema         → 纯文本，直接返回 str 或调用 TextOutputProcessor
├── ImageOutputSchema        → 图像输出（BinaryImage）
├── StructuredTextOutputSchema (ABC)
│   ├── NativeOutputSchema   → output_mode='native'
│   └── PromptedOutputSchema → output_mode='prompted'
└── ToolOutputSchema         → 通过工具调用返回结构化数据（最常用）
```

### OutputSchema.build 的决策流程

```
OutputSpec → OutputSchema.build()
│
├── 包含 NativeOutput?
│   └── → NativeOutputSchema（exclusive，不能与其他类型混用）
│
├── 其他情况，分类 outputs:
│   ├── text_outputs: [str, TextOutput(func)]
│   ├── tool_outputs: [ToolOutput(MyModel)]
│   └── 其他类型: [MyModel, TypedDict, BinaryImage...]
│
├── 只有 str 或 TextOutput?
│   └── → TextOutputSchema
│
├── 只有 BinaryImage?
│   └── → ImageOutputSchema
│
├── 否则（包含结构化类型）?
│   └── → ToolOutputSchema（或 AutoOutputSchema）
│       （区别：是否有 text_output，是否有 DeferredToolRequests）
```

## 4.4 OutputValidator — 输出验证器

```python
OutputValidatorFunc = (
    Callable[[RunContext[AgentDepsT], OutputDataT], OutputDataT | Awaitable[OutputDataT]]
    | Callable[[OutputDataT], OutputDataT | Awaitable[OutputDataT]]
)

@agent.output_validator
async def validate_output(ctx: RunContext[Deps], output: MyModel) -> MyModel:
    if not output.is_valid():
        raise ModelRetry("Output is invalid, please try again")
    return output
```

验证器：
- 在工具调用返回后、结果确认前执行
- 失败时抛出 `ModelRetry`（触发模型重试）
- 最终抛出 `UnexpectedModelBehavior`（超过重试次数）
- 支持同步/异步，有/无 RunContext 四种形式

## 4.5 OutputProcessor — 结构化输出处理器

内部的 `ObjectOutputProcessor` 负责将 LLM 的 JSON 字符串转换为 Python 对象：

```
ToolCallPart（JSON args）
  ↓
args_validator.validate_json()  → Pydantic 验证
  ↓
ObjectOutputProcessor.validate()
  ↓
OutputValidator（用户自定义验证器）
  ↓
FinalResult(output=MyModel)
```

**多类型输出（Union）的处理**：

`UnionOutputProcessor` 和 `UnionOutputModel` 将多个输出类型封装为 Pydantic discriminated union：

```python
# 用户定义
output_type=[ToolOutput(Fruit), ToolOutput(Vehicle)]

# 内部生成的 Union 工具
# 每种类型对应一个工具：return_fruit(name, color), return_vehicle(name, wheels)
# 或合并为一个工具：final_result 接受 Fruit | Vehicle
```

## 4.6 UsageLimits — Token 使用限制

```python
from pydantic_ai.usage import UsageLimits

agent = Agent(
    'openai:gpt-5.2',
    usage_limits=UsageLimits(
        request_limit=10,        # 最多 10 次模型请求
        total_tokens_limit=50000, # 总 token 限制
        input_tokens_limit=30000, # 输入 token 限制
        output_tokens_limit=20000, # 输出 token 限制
        count_tokens_before_request=True,  # 请求前预估 token
    )
)
```

**检查时机**：
- 每次 `ModelRequestNode.run()` 前检查 `request_limit`
- 每次模型响应后检查 token 限制
- `count_tokens_before_request=True` 时：发送请求前调用 `model.count_tokens()`（如果模型支持）

## 4.7 RunUsage / RequestUsage — Token 统计

```python
@dataclass
class UsageBase:
    input_tokens: int        # 输入/提示 token
    output_tokens: int       # 输出/完成 token
    cache_write_tokens: int  # 缓存写入 token（Anthropic prompt caching）
    cache_read_tokens: int   # 缓存读取 token
    input_audio_tokens: int  # 音频输入 token
    output_audio_tokens: int # 音频输出 token
    details: dict[str, int]  # 额外的模型特有统计

@dataclass
class RequestUsage(UsageBase):
    """单次请求的 token 用量。"""

@dataclass
class RunUsage(UsageBase):
    """整个 Agent 运行的累计 token 用量。"""
    requests: int = 0        # 总请求次数

    def incr(self, usage: RequestUsage) -> None:
        """累加单次请求用量。"""
        self.requests += 1
        self.input_tokens += usage.input_tokens
        # ...

    def calculate_price(self) -> PriceCalculation | None:
        """使用 genai-prices 计算费用（需要知道模型名）。"""
```

**`opentelemetry_attributes()`**：将 token 统计转为 OTel 标准属性格式（`gen_ai.usage.input_tokens` 等）。

## 4.8 AgentRunResult — 运行结果（run.py）

```python
@dataclass
class AgentRunResult(Generic[OutputDataT]):
    output: OutputDataT                  # 最终输出
    _output_tool_name: str | None        # 输出工具名（用于对话继续）
    _state: GraphAgentState              # 完整执行状态
    _new_message_index: int              # 本次 run 新增消息的起始索引
    _traceparent_value: str | None       # OTel Trace Parent（用于前端关联）

    # 消息访问
    def all_messages(self, output_tool_return_content=None) -> list[ModelMessage]:
        """获取完整消息历史，可选修改输出工具的返回内容。"""

    def new_messages(...) -> list[ModelMessage]:
        """仅获取本次 run 产生的新消息（用于多轮对话追加）。"""

    def all_messages_json(...) -> bytes:
        """JSON 序列化（用于持久化多轮对话历史）。"""

    # 统计
    def usage(self) -> RunUsage: ...
    def timestamp(self) -> datetime: ...

    # 多轮对话辅助
    def _set_output_tool_return(self, return_content: str) -> list[ModelMessage]:
        """修改输出工具的返回内容，用于继续对话时告知模型工具结果。"""
```

**`new_messages()` 的作用**：多轮对话时，只需将新消息传入下一轮，不必重传全部历史：

```python
result1 = await agent.run("First prompt")
result2 = await agent.run(
    "Second prompt",
    message_history=result1.all_messages()  # 传入完整历史
)
print(result2.new_messages())  # 只看本轮新增的消息
```

## 4.9 FinalResult — 图执行的最终结果

```python
@dataclass
class FinalResult(Generic[OutputDataT]):
    output: OutputDataT
    tool_name: str | None = None      # 触发结束的工具名
    tool_call_id: str | None = None   # 工具调用 ID（用于追踪）
    traceparent: str | None = None    # OTel traceparent
```

`FinalResult` 是 `pydantic_graph` 的 `End` 节点的数据载荷，由 `SetFinalResult` 节点包装后传递给图引擎。

## 4.10 retries.py — HTTP 层重试（与 Agent 重试不同）

`retries.py` 中的 `TenacityTransport`/`AsyncTenacityTransport` 是 **HTTP 传输层的重试**，与 Agent 内的 `ModelRetry` 是不同层次：

| 层次 | 机制 | 触发条件 |
|------|------|---------|
| HTTP 传输层 | `TenacityTransport` | 网络错误、5xx 状态码 |
| Agent 逻辑层 | `ModelRetry` 异常 | 工具失败、输出验证失败 |
| Fallback 层 | `FallbackModel` | `ModelAPIError` 等 |

**`wait_retry_after`**：解析 HTTP `Retry-After` 头，尊重服务端指定的等待时间（秒数或 HTTP 日期格式），超过 max_wait 时截断。

## 总结

```
关键设计决策
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. OutputSpec 的灵活性：
   支持从 str 到复杂 Union 类型，框架自动选择最佳输出模式

2. 工厂模式（OutputSchema.build）：
   用户只需声明 output_type，框架根据类型和模型能力
   自动选择 TextOutputSchema / ToolOutputSchema / NativeOutputSchema

3. OutputValidator：
   在框架层提供最后一道验证关卡，失败时触发模型重试
   而非直接抛出异常给用户

4. Token 预算控制（UsageLimits）：
   在请求前后双重检查，支持预估（count_tokens_before_request）
   避免意外的高成本请求

5. AgentRunResult 的消息管理：
   all_messages vs new_messages 的设计专为多轮对话优化
   output_tool_return_content 允许在继续对话前注入工具结果
```
