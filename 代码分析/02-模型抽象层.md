## 2.2 Model 抽象基类接口

**文件**：`models/__init__.py`（658行起）

### 必须实现的接口

```python
class Model(ABC):
    @abstractmethod
    async def request(
        self,
        messages: list[ModelMessage],
        model_settings: ModelSettings | None,
        model_request_parameters: ModelRequestParameters,
    ) -> ModelResponse:
        """非流式请求。最终由 ModelRequestNode._make_request() 调用。"""

    @abstractmethod
    @property
    def model_name(self) -> str: ...

    @abstractmethod
    @property
    def system(self) -> str:
        """提供商标识符，如 'openai'、'anthropic'。"""

    @cached_property
    @abstractmethod
    def profile(self) -> ModelProfile:
        """模型能力声明。"""
```

### 可选实现（有默认行为）

```python
    @asynccontextmanager
    async def request_stream(...) -> AsyncIterator[StreamedResponse]:
        """流式请求，不实现则不支持流式调用。"""
        raise NotImplementedError(...)

    async def count_tokens(...) -> RequestUsage:
        """Token 预估（用于 UsageLimits.count_tokens_before_request）。"""
        raise NotImplementedError(...)

    def customize_request_parameters(self, params: ModelRequestParameters) -> ModelRequestParameters:
        """自定义请求参数（默认应用 JSON Schema Transformer）。"""
```

### prepare_request — 通用请求预处理（关键方法）

`prepare_request()` 在每次请求前被调用，执行以下逻辑：

```
prepare_request()
├── 合并 model.settings 和运行时 model_settings
├── 调用 customize_request_parameters()（应用 JSON Schema Transformer）
├── 解析 thinking 设置（如果模型支持）
├── 去重内置工具（builtin_tools）
├── 处理 output_mode = 'auto'（根据 Profile 选择最佳输出模式）
├── 重置不相关的输出字段
│     ├── output_mode != 'tool' → 清除 output_tools
│     └── output_mode != 'native'/'prompted' → 清除 output_object
├── 设置 prompted_output_template（如果需要）
├── 将 prompted_output_instructions 追加到 instruction_parts
└── 验证输出模式是否被模型支持
      ├── native output → 检查 profile.supports_json_schema_output
      ├── tool output → 检查 profile.supports_tools
      └── image output → 检查 profile.supports_image_output
```

## 2.3 ModelRequestParameters — 请求配置

```python
@dataclass(repr=False, kw_only=True)
class ModelRequestParameters:
    """传递给模型的请求配置（工具和输出处理相关）。"""

    function_tools: list[ToolDefinition]     # 函数工具定义列表
    builtin_tools: list[AbstractBuiltinTool] # 内置工具（WebSearch、ImageGen等）

    output_mode: OutputMode = 'text'          # 'text'|'tool'|'native'|'prompted'|'auto'
    output_object: OutputObjectDefinition | None
    output_tools: list[ToolDefinition]        # 输出工具（结构化输出时使用）
    prompted_output_template: str | None      # 提示式输出模板
    allow_text_output: bool = True
    allow_image_output: bool = False

    instruction_parts: list[InstructionPart] | None  # 静态/动态指令部分
    thinking: ThinkingLevel | None            # Reasoning 配置
```

**输出模式说明**：

| 模式 | 含义 | 典型场景 |
|------|------|---------|
| `'text'` | 纯文本输出 | 默认，output_type=str |
| `'tool'` | 通过工具调用返回结构化输出 | 大多数有结构化输出的模型 |
| `'native'` | 模型原生 JSON Schema 约束 | OpenAI Structured Outputs |
| `'prompted'` | 通过系统提示要求 JSON 输出 | 不支持 native/tool 的模型 |
| `'auto'` | 根据 Profile 自动选择最佳模式 | 推荐使用，让框架决定 |

## 2.4 ModelProfile — 模型能力声明

**文件**：`profiles/__init__.py`

`ModelProfile` 是模型能力的"说明书"，与具体的 Provider 实现无关。

```python
@dataclass(kw_only=True)
class ModelProfile:
    supports_tools: bool = True                            # 支持工具调用
    supports_json_schema_output: bool = False              # 支持原生 JSON Schema 输出
    supports_json_object_output: bool = False              # 支持 JSON 模式
    supports_image_output: bool = False                    # 支持图像输出
    default_structured_output_mode: StructuredOutputMode   # 默认结构化输出模式
    prompted_output_template: str                          # 提示模板

    supports_thinking: bool = False                        # 支持 thinking/reasoning 配置
    thinking_always_enabled: bool = False                  # 是否总是在 think（o 系列、R1）
    thinking_tags: tuple[str, str] = ('<think>', '</think>')  # Thinking 标签对

    json_schema_transformer: type[JsonSchemaTransformer] | None  # JSON Schema 转换器
    supported_builtin_tools: frozenset[type[AbstractBuiltinTool]] # 支持的内置工具
    ignore_streamed_leading_whitespace: bool               # 忽略流式前导空白
```

**Profile 的设计哲学**：
- Profile 描述"这个模型/系列模型能做什么"
- 与 Provider（认证、连接）和 Model（具体实现）分离
- 通过 `update()` 支持继承式修改：子类 Profile 可以基于父类修改特定字段

**各厂商 Profile 示例**（`profiles/openai.py`）：

```python
class OpenAIModelProfile(ModelProfile):
    # OpenAI 特有的设置
    openai_responses_tool_choice: ...
    system_prompt_role: OpenAISystemPromptRole = 'system'

# 不同 gpt 系列的能力声明
def openai_model_profile(model_name: str) -> OpenAIModelProfile:
    return OpenAIModelProfile(
        supports_json_schema_output=True,        # 支持 Structured Outputs
        supports_thinking=supports_reasoning,    # o 系列支持 thinking
        # ...
    )
```

## 2.5 OpenAI 模型实现（参考实现）

`OpenAIChatModel` 是最完整的实现参考：

### 错误映射
```python
@contextmanager
def _map_api_errors(model_name: str) -> Iterator[None]:
    try:
        yield
    except APIStatusError as e:
        if e.status_code >= 400:
            raise ModelHTTPError(status_code=...) from e
    except APIConnectionError as e:
        raise ModelAPIError(message=...) from e
```

所有模型实现都应将 SDK 特定的异常转换为 pydantic-ai 的标准异常（`ModelHTTPError`、`ModelAPIError`），使 `FallbackModel` 的错误检测逻辑正常工作。

- `OpenAIChatModel`：Chat Completions API（`/v1/chat/completions`）

## 2.6 FallbackModel — 故障转移模型

**文件**：`models/fallback.py`

`FallbackModel` 是装饰器模式的典型应用，将多个模型串成 Fallback 链：

```python
model = FallbackModel(
    'openai:gpt-5.2',           # 首选
    'anthropic:claude-opus-4',  # 备选1
    'google:gemini-2.0-flash',  # 备选2
    fallback_on=(ModelAPIError, RateLimitError),  # 何时触发 fallback
)
```

**Fallback 触发条件**（`fallback_on` 参数）：

| 类型 | 示例 | 说明 |
|------|------|------|
| 异常类型 | `ModelAPIError` | 抛出该类型异常时触发 |
| 异常元组 | `(ModelAPIError, TimeoutError)` | 任一异常触发 |
| 异常处理函数 | `lambda e: isinstance(e, MyErr)` | 自定义判断逻辑 |
| 响应处理函数 | `def check(r: ModelResponse) -> bool` | 基于响应内容判断（空回复等） |
| 混合序列 | `[ModelAPIError, check_response]` | 组合多种条件 |

**响应处理函数的类型检测**：
通过检查函数第一个参数的类型注解来区分异常处理函数和响应处理函数：
```python
def _is_response_handler(handler):
    first_param_type = get_first_param_type(handler)
    return first_param_type is ModelResponse  # 精确匹配
```

**流式 Fallback**：流式场景使用 `AsyncExitStack` 管理 `request_stream()` 上下文，在失败时自动清理资源。

## 2.7 WrapperModel — 透明代理基类

**文件**：`models/wrapper.py`

`WrapperModel` 是所有包装器模型的基类，通过转发所有方法实现透明代理：

```python
class WrapperModel(Model):
    wrapped: Model  # 被包装的模型

    async def request(self, ...) -> ModelResponse:
        return await self.wrapped.request(...)

    async def request_stream(self, ...) -> AsyncIterator[StreamedResponse]:
        async with self.wrapped.request_stream(...) as stream:
            yield stream

    # 其他方法同样转发...
```

子类只需覆盖想要拦截的方法即可，其余方法自动透传。

**`CompletedStreamedResponse`**（在 wrapper.py 中定义）：
将已完成的 `ModelResponse` 包装为 `StreamedResponse`，供 durable execution 集成使用（Temporal、DBOS 等在 Activity/Task 中消费流式响应，返回最终结果）。

## 2.8 InstrumentedModel — OTel 追踪包装

**文件**：`models/instrumented.py`

无侵入地为任意 Model 添加 OpenTelemetry 追踪：

```python
@dataclass
class InstrumentedModel(WrapperModel):
    """透明代理，拦截 request/request_stream，创建 OTel Span。"""

    settings: InstrumentationSettings  # 追踪选项
```

**追踪的内容**：
- **Span 属性**：`gen_ai.system`、`gen_ai.request.model`、`gen_ai.request.temperature`、`gen_ai.usage.input_tokens` 等
- **Events/Logs**（依 `event_mode` 而定）：消息内容（可配置是否包含）
- **Token Histogram**：按 OTel Gen AI 语义约定发布 token 用量直方图

**`InstrumentationSettings`**：

```python
@dataclass
class InstrumentationSettings:
    tracer: Tracer
    logger: Logger
    event_mode: Literal['attributes', 'logs'] = 'attributes'  # 事件记录方式
    include_binary_content: bool = True   # 是否记录二进制内容
    include_content: bool = True           # 是否记录消息文本
    version: int = DEFAULT_INSTRUMENTATION_VERSION
```

## 2.10 StreamedResponse — 流式响应抽象

```python
class StreamedResponse(ABC):
    model_request_parameters: ModelRequestParameters

    @abstractmethod
    async def _get_event_iterator(self) -> AsyncIterator[ModelResponseStreamEvent]:
        """模型特定的事件生成器（子类实现）。"""

    def __aiter__(self) -> AsyncIterator[ModelResponseStreamEvent]:
        """公开的异步迭代接口。"""

    def get(self) -> ModelResponse:
        """流式完成后获取完整 ModelResponse。"""

    @property
    @abstractmethod
    def model_name(self) -> str: ...

    @property
    @abstractmethod
    def timestamp(self) -> datetime: ...
```

`StreamedResponse` 内部有一个 `_ModelResponsePartsManager`，将流式事件累积成最终 `ModelResponse`。

## 2.11 infer_model — 模型实例化入口

`models/__init__.py` 中的 `infer_model()` 函数是统一的模型实例化入口：

```python
def infer_model(model: Model | KnownModelName | str) -> Model:
    """从字符串/实例创建 Model。

    字符串格式：'provider:model-name'
    示例：
        'openai:gpt-5.2'      → OpenAIChatModel('gpt-5.2')
        'anthropic:claude-...' → AnthropicModel('claude-...')
        'google:gemini-...'   → GoogleModel('gemini-...')
    """
```

字符串解析流程：
1. 按 `:` 分割 provider 和 model_name
2. 通过 `infer_provider_class()` 找到对应的 Provider 类
3. Provider 类创建对应的 Model 实例

## 总结

```
关键设计决策
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 三层分离：
   - Model（实现）：负责 API 调用和格式转换
   - ModelProfile（能力）：声明模型支持什么
   - Provider（配置）：处理认证和连接

2. prepare_request 通用化：
   父类 Model.prepare_request() 处理所有通用逻辑（思考设置、
   输出模式选择、内置工具去重），子类只管调用 API

3. 装饰器模式（WrapperModel）：
   FallbackModel、InstrumentedModel 无需知道底层实现
   即可添加 Fallback / OTel 追踪

4. 异常标准化：
   所有 Model 实现都将 SDK 异常转换为 ModelHTTPError/ModelAPIError，
   使上层逻辑（FallbackModel、重试）可以统一处理

5. 输出模式由 Profile 决定：
   'auto' 模式让框架根据 Profile.default_structured_output_mode
   自动选择最优输出策略
```
